# 07. 제어기 검증 체크리스트

제어기 설명은 파형 한 장으로 끝나지 않는다. 어떤 조건에서 무엇을 통과해야 하는지
재현할 수 있어야 포트폴리오의 증거가 된다.

## 1. 요구사항

- [ ] 제어 대상, 입력, 출력, 단위를 정의했다.
- [ ] 목표 범위와 동작 모드를 정의했다.
- [ ] 상승시간, 정착시간, 오버슈트, 정상상태 오차 기준이 있다.
- [ ] 출력 포화, slew-rate, 전류·전압 제한을 정의했다.
- [ ] fault와 보호 로직의 우선순위를 정의했다.

## 2. 모델

- [ ] 모델의 동작점과 유효 범위를 기록했다.
- [ ] 파라미터 출처와 허용 편차가 있다.
- [ ] 센서 필터, 계산 지연, PWM/샘플링을 포함했다.
- [ ] 연속시간과 이산시간 모델을 비교했다.
- [ ] 모델을 만들 때 쓰지 않은 별도 데이터로 검증했다.

## 3. 기본 시험 시나리오

| 시험 | 자극 | 확인 결과 |
|---|---|---|
| 목표값 추종 | 최소·정격·최대 지령 step/ramp | 응답시간, 오버슈트, 오차 |
| 부하 외란 | 부하 증가·감소 | 최대 편차, 회복시간 |
| 입력 외란 | 입력전압 또는 주파수 변화 | 안정성, 제한 진입 |
| 센서 잡음 | 백색잡음·리플·양자화 | 출력 리플, D항 민감도 |
| 포화 | 큰 지령과 장시간 제한 | windup, 회복시간 |
| 파라미터 편차 | L/C/R, gain, 지연 변화 | 여유와 worst case |
| 모드 전환 | enable/disable, 방향·모드 변경 | bumpless transfer |
| fault | 센서 오류, 과전류 등 | 안전 출력과 reset 조건 |

## 4. 단계별 증거

```text
Model review
  → MIL: 알고리즘과 요구사항 검증
  → SIL/PIL: 코드·수치형·실행주기 검증
  → HIL: I/O, 지연, fault, 보호 검증
  → Bench: 실제 하드웨어·센서·전력단 검증
```

각 단계에서 다음을 남긴다.

- 모델·코드 버전과 commit
- 파라미터 세트와 시험환경
- 입력 시나리오와 자동 판정식
- 원시 결과와 요약 그래프
- Pass/Fail 및 실패 원인
- 변경 전후 비교

## 5. 포트폴리오 공개 기준

- 회사명, 고객명, 회로 상세, 비공개 calibration 값은 익명화한다.
- 공개할 수 없는 실제 수치 대신 정규화 값이나 재현용 공개 모델을 사용한다.
- `문제 → 접근 → 검증 → 결과`가 한 화면에서 연결되도록 요약한다.
- 수행하지 않은 시험은 `계획`, `미수행`, `TBD`로 명확히 구분한다.

## 완료 정의

아래 네 질문에 모두 답할 수 있을 때 해당 제어 주제를 완료로 본다.

1. 어떤 문제를 해결했는가?
2. 왜 이 구조와 gain을 선택했는가?
3. 어떤 조건에서 통과·실패했는가?
4. 다른 사람이 같은 결과를 재현할 수 있는가?
